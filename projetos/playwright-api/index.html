// problema que eu precisava resolver

Por que eu organizei o framework desta forma

Meu objetivo era reduzir repetição sem esconder o comportamento dos testes. Eu precisava permitir execução paralela, reutilizar autenticação, criar dados claros e deixar a falha explicável no relatório.

01

Contexto isolado

Eu dei a cada fixture seu próprio APIRequestContext. O token podia ser compartilhado, mas o recurso descartável não, evitando que uma execução encerrasse o contexto de outra.

02

Dados expressivos

Eu usei builders com valores válidos por padrão para que o cenário mostrasse somente a alteração responsável pela condição testada.

03

Asserts reutilizáveis

Eu centralizei validações HTTP, JSON e GraphQL para manter mensagens de falha consistentes e reduzir código repetido.

04

Diagnóstico visível

Eu levei etapas, metadados, requests e responses ao Allure para reduzir o tempo entre a falha e a identificação da causa.
